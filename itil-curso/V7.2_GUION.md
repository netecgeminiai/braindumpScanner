### [00:08-01:00] INTRODUCCIÓN Y PROPÓSITO
**[VISUAL: Título "Desarrollo y Gestión de Software". Icono de código fuente y engranajes]**

Hola y bienvenidos. En un mundo donde todo es digital, la capacidad de una organización para crear software propio o personalizar software comprado se ha convertido en una ventaja competitiva fundamental.

Hablemos de la **Práctica de Desarrollo y Gestión de Software**.

El propósito de esta práctica es asegurar que las aplicaciones cumplan con las necesidades de las partes interesadas internas y externas en términos de funcionalidad, confiabilidad, mantenibilidad, cumplimiento normativo y auditabilidad.

En palabras simples: hacer software que funcione, que sea seguro y que la gente realmente quiera usar.

---

### [01:00-02:15] EL CICLO DE VIDA DEL SOFTWARE Y LOS ENFOQUES
**[VISUAL: Pantalla dividida. Izquierda: Cascada (Waterfall) cayendo bloque por bloque. Derecha: Ciclo ágil (Sprint/Loop infinito)]**

Históricamente, el software se construía usando un enfoque en "Cascada" (Waterfall). Primero se diseñaba todo el sistema, luego se programaba todo, luego se probaba todo y, dos años después, se lanzaba. ¿El problema? A menudo el mercado ya había cambiado.

Hoy en día, ITIL (versión 5) reconoce que el mundo requiere enfoques más flexibles. Hablamos del **Desarrollo Ágil de Software (Agile)**.

En Agile, el trabajo se divide en pequeños bloques o iteraciones (a menudo llamados "Sprints"). En lugar de esperar dos años, el equipo entrega una pequeña parte funcional del software en dos semanas, obtiene retroalimentación del cliente y ajusta el rumbo. Esto se alinea perfectamente con el principio guía de ITIL: *"Progresar iterativamente con retroalimentación"*.

---

### [02:15-03:15] LA GESTIÓN CONTINUA (MANTENIBILIDAD)
**[VISUAL: Un coche en el taller mecánico recibiendo mantenimiento continuo]**

Pero atención a la segunda parte del nombre de esta práctica: Desarrollo *y Gestión* de Software. 

El trabajo de un desarrollador no termina el día que la aplicación sale a la luz. De hecho, ahí es cuando empiezan los verdaderos costos. 

El software necesita mantenimiento constante. Necesita parches de seguridad, actualizaciones para nuevas versiones de sistemas operativos, y arreglos de *bugs* o errores. Si el software se programó de forma desordenada y rápida, será una pesadilla mantenerlo. Esta práctica asegura que el código esté bien documentado y estructurado para ser "mantenible" a largo plazo.

---

### [03:15-04:00] INFRAESTRUCTURA COMO CÓDIGO (IaC) Y DEVOPS
**[VISUAL: Logo o concepto de DevOps (Infinito) e Infraestructura en la nube (AWS/Azure)]**

En la era moderna, esta práctica también adopta el concepto de **DevOps** (la unión de Desarrollo y Operaciones de TI).

Hoy, no solo las aplicaciones se escriben con código. Gracias a la nube, los servidores y las redes enteras se configuran usando programación. Esto se conoce como **Infraestructura como Código**. 

Esto permite que, si necesitamos un nuevo servidor para probar nuestra aplicación, no tenemos que pedirlo y esperar una semana a que llegue por correo; simplemente ejecutamos un script y Amazon o Azure nos levantan un servidor virtual en 30 segundos.

**[VISUAL: Resumen de la práctica: Agile, Mantenibilidad, DevOps]**

En resumen, la práctica de Desarrollo y Gestión de Software diseña, programa y mantiene las aplicaciones del negocio utilizando metodologías modernas y ágiles para asegurar que el código entregue valor real y sostenible.
